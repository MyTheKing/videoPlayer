# BMAD - Epic Breakdown

## Overview

本文档提供 BMAD 本地视频播放器项目的完整 Epic 与 Story 拆解，将 PRD、架构文档中的需求分解为可实施的故事。

## Requirements Inventory

### Functional Requirements

- FR1: 用户能扫描设备本地存储中的视频文件并纳入应用可见列表。
- FR2: 用户能以卡片式浏览已扫描到的本地视频。
- FR3: 用户能按类型、大小、日期等条件筛选/查看本地视频。
- FR4: 用户能多选本地视频并执行批量添加或删除操作。
- FR5: 用户能触发重新扫描以更新本地视频列表。
- FR6: 用户能创建播放列表/歌单并为之命名。
- FR7: 用户能先建空歌单再批量添加视频。
- FR8: 用户能把选中的本地视频添加进指定歌单。
- FR9: 用户能对歌单内视频进行拖拽排序。
- FR10: 用户能对歌单内视频进行批量移除或批量操作。
- FR11: 用户能为歌单设置或更换封面（自定义封面）。
- FR12: 用户能按歌单名称或内容快速找到某个歌单。
- FR13: 用户能播放、暂停、上一首、下一首当前歌单中的视频。
- FR14: 用户能选择单曲循环、列表循环或随机播放模式。
- FR15: 用户能在息屏或切到其他应用时继续听当前视频的音频（后台播放）。
- FR16: 用户能通过通知栏控制播放/暂停、上一首/下一首（酷狗/QQ 音乐风格）。
- FR17: 用户能在播放界面使用手势控制（如滑动切歌、双击暂停等）。
- FR18: 用户能将视频标记为收藏（与「喜欢」合并为同一能力）。
- FR19: 用户能为视频添加多个标签并进行管理。
- FR20: 用户能在列表视图与标签视图之间切换以浏览内容。
- FR21: 用户能按标签、歌单、收藏、历史等维度筛选或进入相应内容列表。
- FR22: 用户能按视频标题进行模糊搜索。
- FR23: 用户能查看与使用历史搜索记录。
- FR24: 用户能在搜索结果中看到与关键词匹配的高亮。
- FR25: 用户能在应用内正常观看视频画面。
- FR26: 用户能调节播放倍速（如 0.5x～2x）。
- FR27: 用户能使用画中画模式观看视频。
- FR28: 用户能设置睡眠定时（定时停止播放）。
- FR29: 用户能查看最近播放、继续观看、最近添加等入口并从中进入播放。
- FR30: 用户能关联本地歌词文件与视频并保存该关联（路径）。
- FR31: 用户能在播放时查看与当前视频关联的歌词内容。
- FR32: 用户能查看包含图片与文字等富文本格式的歌词。
- FR33: 用户能查看视频的时长、分辨率、文件大小等基础信息。
- FR34: 用户能以卡片式浏览视频时看到上述基础信息或缩略图。
- FR35: 用户能在播放控制区使用悬浮式控制按钮（或等效入口）。
- FR36: 系统能持久化播放列表、歌单、收藏、标签、播放历史等数据（本地）。
- FR37: 用户能依赖本地数据在离线状态下使用上述所有与本地数据相关的能力。
- FR38: 系统能在需要访问本地视频时向用户请求或引导授予存储/媒体相关权限。
- FR39: 系统能在权限不足、路径失效或扫描失败时向用户给出可理解的提示，并在可行时提供重试或重新扫描入口。

### NonFunctional Requirements

- NFR-P1: **Performance - 扫描与列表**：在典型设备上，对≤5000 个本地视频的扫描与列表展示应在用户可接受时间内完成（例如≤30s），且列表滚动无明显卡顿。
- NFR-P2: **Performance - 播放启动**：从用户选择播放到首帧/首声可感知，延迟应控制在较短时间内（例如≤3s，视文件与设备而定）。
- NFR-P3: **Performance - 切歌与通知栏**：上一首/下一首、通知栏控制等操作应在 1s 内生效，避免明显迟滞。
- NFR-R1: **Reliability - 后台与息屏**：在息屏、切到其他应用、接听电话等场景下，音频播放应能持续进行；仅在用户主动停止或系统极端资源策略下才中断。
- NFR-R2: **Reliability - 进程与数据**：应用被系统回收后再次打开，应能恢复上次播放进度（或在合理范围内），且歌单、收藏、标签等本地数据不丢失。
- NFR-S1: **Security & Privacy**：数据仅存于本设备，无账号、无云端；自用场景下以「可直接使用」为前提，不在此阶段细化权限申请或隐私政策；后期若引入服务端或上架再补充合规。
- NFR-O1: **其他**：Scalability/Accessibility/Integration 本阶段不单独设定；若有需求在后续版本补充。

### Additional Requirements

- **Starter / 首条实现**：架构指定以 KotlinDemo（`Z:\FE\Android\KotlinDemo\`）为基线；首条实现类故事为「在 KotlinDemo 中完成依赖配置与最小可运行播放链路」—— 需在 `libs.versions.toml` 与 `app/build.gradle.kts` 中补齐 Media3、Room（含 KSP）、Navigation Compose、ViewModel Compose、Coil 等依赖并实现 MediaSession + ExoPlayer 前台服务、单视频播放、通知栏控制。
- **数据与存储**：Room 存 Playlist、PlaylistItem、VideoMedia、PlayHistory、Tag 等；表/列 snake_case，迁移与 schema 导出到 `schemas/`；Repository/UseCase 层做路径存在性等校验。
- **播放与 UI 边界**：播放控制仅经 MediaController + MediaSession；UI 不直接持 ExoPlayer；ViewModel 暴露 StateFlow/Intent，Compose 消费；按功能包 player/playlist/library/settings 组织，每功能 ui/、可选 domain/。
- **权限与错误**：存储/媒体权限在进入或扫描前请求；权限不足、路径失效、扫描失败时在 UI 给出可理解提示与重试/重新扫描入口；逻辑在 ViewModel，不散落在 UI。
- **文档与一致**：对外暴露的类、接口、函数需写全 KDoc（用途、@param/@return/@throws）；命名、State/Intent、播放边界按架构文档中的 Implementation Patterns 执行。
- **实现顺序（架构）**：1) 依赖与最小播放链路 → 2) Room schema + Repository → 3) 扫描与本地视频列表 → 4) 歌单 CRUD 与加入播放队列 → 5) Compose 主流程界面 + ViewModel + 导航 → 6) 权限申请与失败提示、播放生命周期与进程恢复。

## Epic 1: 首次使用与「播一首」

用户能安装打开 APP、授予存储/媒体权限、播出一个本地视频（从最小入口选文件或极简扫描）、在息屏/切应用时继续听，并用通知栏控制播放/暂停、上一首/下一首。

### Story 1.1: KotlinDemo 依赖配置与最小可运行环境

As a 开发者,
I want 在 KotlinDemo 中补齐 Media3、Room(KSP)、Navigation、ViewModel Compose、Coil 等依赖并 Sync 通过,
So that 工程可编译运行，为后续播放与数据层实现打好基础。

**Acceptance Criteria:**

**Given** 已存在 KotlinDemo 项目（`Z:\FE\Android\KotlinDemo\`）  
**When** 在 `libs.versions.toml` 与 `app/build.gradle.kts` 中按要求添加 Media3（exoplayer/session/ui）、Room（runtime/ktx/compiler + KSP）、Navigation Compose、lifecycle-viewmodel-compose、Coil Compose，并修正 `compileSdk` 等配置  
**Then** `./gradlew assembleDebug` 或 Android Studio Sync 通过，无编译错误  
**And** 依赖版本在 version catalog 中统一管理

### Story 1.2: 存储/媒体权限请求与引导

As a 用户,
I want 在首次进入或需要访问本地视频时被请求存储/媒体权限，拒绝时看到可理解的说明与「去设置」入口,
So that 我能知道为什么需要权限并自行前往设置开启。

**Acceptance Criteria:**

**Given** 应用已安装并首次打开（或尚未授予存储/媒体权限）  
**When** 进入需访问本地视频的流程（如扫描或选文件）  
**Then** 系统请求 READ_MEDIA_VIDEO / READ_EXTERNAL_STORAGE 等（按 targetSdk 选用）  
**And** 若用户拒绝，展示可理解的文案与「去设置」入口，不崩溃

### Story 1.3: 单视频播放与 MediaSession 前台服务

As a 用户,
I want 从最小入口（如选一个本地文件或极简列表点一条）选择视频后能播出声画，息屏或切到其他应用时继续听,
So that 我能先「播一首」验证播放与后台能力。

**Acceptance Criteria:**

**Given** 已具有存储/媒体权限  
**When** 用户通过最小入口选到一个本地视频并点击播放  
**Then** 使用 MediaSession + ExoPlayer 前台服务播放该视频，有声有画  
**And** 息屏或切到其他应用时音频继续播放（NFR-R1 相关）

### Story 1.4: 通知栏控制（酷狗/QQ 风格）

As a 用户,
I want 播放时在通知栏看到播放控制，并能控制播放/暂停、上一首/下一首,
So that 不需亮屏即可操作播放。

**Acceptance Criteria:**

**Given** 当前正在播放  
**When** 用户下拉通知栏  
**Then** 展示酷狗/QQ 风格媒体通知，含封面/标题、播放/暂停、上一首/下一首  
**And** 操作在 1s 内生效（NFR-P3）

### Story 1.5: 单曲/列表/随机循环模式

As a 用户,
I want 能切换单曲循环、列表循环、随机播放模式,
So that 播放顺序符合我的习惯。

**Acceptance Criteria:**

**Given** 当前有播放列表或单条媒体  
**When** 用户切换循环模式（单曲/列表/随机）  
**Then** 行为经 MediaController/ControlDispatcher 生效，与架构约定一致  
**And** 模式在本次会话内保持，直至用户再次切换

### Story 1.6: 权限不足/路径失效/扫描失败提示与重试

As a 用户,
I want 在权限不足、路径失效或扫描失败时看到可理解的提示，并在可行时能重试或重新扫描,
So that 我知道出了什么问题并有机会自行恢复。

**Acceptance Criteria:**

**Given** 出现权限不足、路径失效或扫描失败等情况  
**When** 系统检测到上述情况  
**Then** 在 UI 展示可理解的文案（非原始异常堆栈）  
**And** 在可行时提供「重试」或「重新扫描」等入口（FR39）

## Epic 2: 本地视频库

用户能扫描设备本地视频、以卡片形式浏览、按类型/大小/日期筛选、多选并批量添加或删除、查看时长/分辨率/大小与缩略图、触发重新扫描。

### Story 2.1: 本地视频扫描与入库

As a 用户,
I want 能触发扫描设备本地存储中的视频文件，结果纳入应用可见列表,
So that 我能在应用里看到本机的视频。

**Acceptance Criteria:**

**Given** 已授予存储/媒体权限  
**When** 用户触发「扫描」或首次进入本地库  
**Then** 应用扫描约定或可选目录下的视频文件并纳入可见列表  
**And** 典型设备上 ≤5000 个视频的扫描在可接受时间内完成（如 ≤30s）（NFR-P1）

### Story 2.2: 卡片式浏览与基础信息/缩略图

As a 用户,
I want 以卡片形式浏览已扫描视频，并看到时长、分辨率、文件大小及缩略图,
So that 我能快速辨认并选择要播的视频。

**Acceptance Criteria:**

**Given** 已完成至少一次扫描且有本地视频  
**When** 用户进入本地视频库  
**Then** 以卡片形式展示列表，卡片含缩略图、时长、分辨率、文件大小等（FR33, FR34）  
**And** 列表滚动无明显卡顿（NFR-P1）

### Story 2.3: 按类型/大小/日期筛选

As a 用户,
I want 能按类型、大小、日期等条件筛选或查看本地视频,
So that 我能缩小范围快速找到目标。

**Acceptance Criteria:**

**Given** 本地视频列表已展示  
**When** 用户选择筛选条件（类型/大小/日期等）  
**Then** 列表仅展示符合条件的视频  
**And** 可清除筛选恢复全部

### Story 2.4: 多选与批量添加/删除

As a 用户,
I want 能多选本地视频并执行批量添加（如加入歌单）或批量删除（从列表移除）,
So that 我能成批管理视频。

**Acceptance Criteria:**

**Given** 本地视频列表已展示  
**When** 用户进入多选模式并勾选若干条后执行「批量添加」或「批量删除」  
**Then** 批量添加时能将所选视频加入指定歌单（若已实现歌单）；批量删除时从当前可见列表中移除或取消勾选  
**And** 操作结果即时反映在列表上

### Story 2.5: 重新扫描

As a 用户,
I want 能主动触发重新扫描以更新本地视频列表,
So that 新下载或新产生的视频能出现在应用里。

**Acceptance Criteria:**

**Given** 用户已在本地视频库或设置中  
**When** 用户触发「重新扫描」  
**Then** 再次执行扫描逻辑并更新可见列表  
**And** 若扫描失败，按 Story 1.6 给出提示与重试入口（FR5）

### Story 3.1: 创建歌单并命名

As a 用户,
I want 能创建新的播放列表/歌单并为之命名,
So that 我能按主题或场景管理视频。

**Acceptance Criteria:**

## Epic 8: 数据持久化与恢复

用户能在关掉再开后仍保留歌单/收藏/标签/历史；进程被系统回收后再次打开，能恢复上次播放进度（或在合理范围内），本地数据不丢失。

### Story 8.1: Room 持久化歌单/收藏/标签/历史

As a 用户,
I want 歌单、收藏、标签、播放历史等关掉应用再打开后仍在,
So that 我的整理和进度不被丢失。

**Acceptance Criteria:**

**Given** 用户已创建歌单、收藏、标签或产生播放历史  
**When** 用户完全退出应用后再次打开  
**Then** 上述数据仍存在并可正常使用  
**And** 通过 Room 持久化，表/列按架构 snake_case，Migration 与 schema 导出（FR36）

### Story 8.2: 离线使用与进程恢复

As a 用户,
I want 在无网络时也能使用全部基于本地数据的能力；进程被系统回收后再次打开，能恢复上次播放进度与本地数据,
So that 离线可用且不丢数据。

**Acceptance Criteria:**

**Given** 无网络或进程曾被系统回收  
**When** 用户再次打开应用  
**Then** 歌单、收藏、标签、历史等均从本地加载，可正常使用（FR37）  
**And** 若存在「上次未播完」的会话，在合理范围内恢复播放进度，本地数据不丢失（NFR-R2）
