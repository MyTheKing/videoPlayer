# Product Requirements Document - BMAD

**Author:** cursorAgent
**Date:** 2026-01-27

---

## Success Criteria

### User Success

- 用户能在锁屏/后台下，用歌单连续听本地视频里的音频，并用通知栏完成播放/暂停/切歌，无需亮屏。
- 用户会创建至少一个歌单、向其中添加本地视频，并在约一周内多次使用「锁屏听」或「通知栏控制」。
- 用户有「不用转格式、不用开屏，就像听歌一样听视频」的感受，并愿意在通勤、运动等场景持续使用。

### Business Success

- **当前**：个人项目，无营收与用户增长 KPI；目标为「做出可用的本地视频播放器，支持锁屏听、歌单与基础管理」。
- **后期可选**：若做分发或轻量商业化，再补下载量、留存、满意度等指标。

### Technical Success

- MVP 功能在本地可完整自测通过。
- 核心路径「扫描本地视频 → 建歌单 → 添加视频 → 锁屏播放 → 通知栏控制」可端到端走通。
- 本地存储与播放架构支持后续扩展（如预留服务端对接）。

### Measurable Outcomes

| 类别       | 指标说明                                                   |
|------------|------------------------------------------------------------|
| 功能完成度 | MVP 功能全部实现并可本地自测通过                           |
| 核心路径   | 「扫描 → 建歌单 → 锁屏播放」可完整走通                     |
| 使用意愿   | 本人能在一周内多次用该 APP 锁屏听视频（可自评）             |

## Product Scope

### MVP - Minimum Viable Product

- 本地视频扫描与添加（卡片式展示、多选、拖拽添加、按添加时间排序）
- 播放列表/歌单（自建、命名、空列表+批量添加、拖拽排序、批量操作）
- 播放控制与息屏（播放/暂停、上一首/下一首、单曲/列表/随机循环、息屏播放、酷狗/QQ 风格通知栏）
- 本地存储（Room：播放列表、收藏、标签、历史；结构预留日后服务端对接）
- 收藏与分类（收藏与喜欢合并、多标签、列表/标签视图切换、按标签/歌单/收藏/历史查找）
- 搜索（按视频标题模糊搜索、历史搜索、搜索高亮）
- 看与听（正常看视频、倍速/画中画/睡眠定时、播放列表封面）
- 歌词（本地歌词文件、路径保存、富文本展示）
- 基础信息与交互（时长/分辨率/大小、卡片式浏览、悬浮控制、手势、最近播放/继续观看/最近添加）

### Growth Features (Post-MVP)

- 新用户引导步骤
- 音频转文字歌词、歌词滚动同步
- 音效（均衡器、3D 环绕等）
- 播放速度记忆、章节、离线下载（若有服务端再考虑）
- 分享、家人共享、多设备同步（后期可选）

### Vision (Future)

- 分享、家人共享、多 profile、分发或轻量商业化；架构预留从纯本地平滑演进。

## User Journeys

### 日常用户（锁屏听）— 主路径

**开场**：用户已有从 YouTube 等渠道下载的赞美诗、课程等视频，想在通勤、走路、骑车时像听歌一样听，但不想亮屏、转格式或导入音乐 APP。

**过程**：安装 APP → 授权存储/媒体 → 扫描本地视频 → 卡片式浏览并多选 → 创建歌单（如「赞美诗」「通勤听」）并添加 → 选歌单播放 → 锁屏或切到其他应用 → 用通知栏完成暂停/上一首/下一首。

**价值时刻**：在锁屏或后台下连续听、用通知栏切歌，体会到「不用转格式、不用开屏，和听歌一样」。

**结局**：通勤、运动等场景下稳定用该 APP 当「听视频」工具，形成习惯。

---

### 日常用户 — 边缘与恢复

**场景**：歌单播完、误删视频、路径失效、权限被回收等。

**过程**：列表播完 → 依赖循环/随机续播或手动再选歌单；误删/路径失效 → 在「本地视频」或「歌单」里发现缺失 → 重新扫描或重新添加；权限被回收 → 首次再进时引导授权。

**需求**：循环/随机可配置、列表可编辑、扫描可重跑、权限引导与错误提示清晰。

---

### 重度用户（多列表、多分类）— 主路径

**开场**：本地视频很多，希望按场景、主题或心情快速找到要听/看的内容。

**过程**：建多个歌单（学习、运动、放松等）→ 为视频打多标签 → 用列表/标签视图切换浏览 → 拖拽排序、批量添加/移除 → 按标题搜索、用历史/收藏/歌单入口快速进入 → 选歌单或收藏项播放，锁屏听或进详情看视频。

**价值时刻**：在大量视频下仍能很快定位到「当前想听的那一类」，并保持歌单与收藏的可维护性。

**结局**：歌单与标签成为主要导航方式，搜索与历史作为补充，形成稳定的「听+看」工作流。

---

### 其他用户类型

- **Admin/Operations**：不适用（纯本地、个人使用）。
- **Support/Troubleshooting**：不适用（无客服/工单；问题靠自检与文档）。
- **API/Integration**：不适用（无对外开放 API）。

## Mobile App Specific Requirements

### Project-Type Overview

- **类型**：原生 Android 应用（Java/Kotlin）
- **形态**：本地视频播放器，支持锁屏听、歌单、通知栏控制
- **部署**：当前自用，不依赖应用商店上架；权限与隐私按「可直接使用」处理，合规与上架流程留到后续有服务端或正式分发时再考虑

### Technical Architecture Considerations

- **平台**：仅支持 Android；采用原生实现，便于使用系统媒体会话、后台播放、存储与权限能力
- **离线**：全离线、纯本地；无账号、无云端，所有数据在设备本地（Room + 媒体文件）
- **架构可扩展**：本地库与「播放/列表/收藏」等抽象预留日后对接服务端的可能，便于从纯本地平滑演进

### Platform Requirements (platform_reqs)

- **最低版本**：按所选播放引擎（如 ExoPlayer）与 MediaSession 要求设定（建议 Android 6.0+ 或按依赖实际要求）
- **后台与媒体**：使用 MediaSession + 前台/媒体服务，保证息屏、切到其他应用时仍可播放与通知栏控制
- **存储**：读本地视频文件、缩略图、歌词文件；写入仅限应用私有目录与 Room 数据库

### Device Permissions & Features (device_permissions)

- **存储/媒体**：需存储/媒体相关权限以扫描与读取本地视频；当前自用，按「能直接使用」前提实现，不在本阶段单独做权限申请流程或隐私政策
- **后台/电池**：后台播放与通知栏控制需系统允许后台运行；必要时在设置中引导用户关闭针对本应用的省电限制
- **不依赖**：GPS、摄像头、麦克风、蓝牙等与本 MVP 无关

### Offline Mode (offline_mode)

- **全程离线**：无登录、无拉流、无云同步；扫描、播放、歌单、收藏、标签、历史、搜索均基于本地数据
- **数据持久化**：Room 存播放列表、歌单、收藏、标签、播放历史；歌词与视频文件路径存于本地

### Push Strategy (push_strategy)

- **当前**：无服务端，无推送；不涉及 push notifications
- **后期**：若引入服务端，再考虑「新内容/歌单更新」等推送策略

### Store Compliance (store_compliance)

- **当前**：自用，不上架；不在此阶段写商店合规或审核要求
- **后期**：若上架 Google Play 或其他渠道，再补充隐私政策、权限说明、目标 API Level、内容政策等

### Implementation Considerations

- **播放**：优先采用现成播放引擎（如 ExoPlayer）与系统 MediaSession，实现解码、进度、息屏与通知栏控制
- **UI**：Material Design；卡片式浏览、悬浮控制、列表/标签视图、手势（滑动切歌、双击暂停等）按既有 PRD/Product Brief 执行
- **组件策略**：优先使用现成库与系统 API（播放、存储、通知栏）；仅在确有缺陷时再自研替换
- **技术栈**：Kotlin/Java、Room、DataStore/SharedPreferences、可选 Hilt/ViewModel 等，按现有技术选型为准

- **资源**：个人开发、自用；以「可直接使用」为前提，不在此阶段做权限申请流程或上架合规。
- **验收**：核心路径「扫描 → 建歌单 → 添加视频 → 锁屏播放 → 通知栏控制」可完整走通，本人能在一周内多次用该 APP 锁屏听视频。

## Functional Requirements

### 本地视频与扫描

- FR1: 用户能扫描设备本地存储中的视频文件并纳入应用可见列表。
- FR2: 用户能以卡片式浏览已扫描到的本地视频。
- FR3: 用户能按类型、大小、日期等条件筛选/查看本地视频。
- FR4: 用户能多选本地视频并执行批量添加或删除操作。
- FR5: 用户能触发重新扫描以更新本地视频列表。

### 播放列表与歌单

- FR6: 用户能创建播放列表/歌单并为之命名。
- FR7: 用户能先建空歌单再批量添加视频。
- FR8: 用户能把选中的本地视频添加进指定歌单。
- FR9: 用户能对歌单内视频进行拖拽排序。
- FR10: 用户能对歌单内视频进行批量移除或批量操作。
- FR11: 用户能为歌单设置或更换封面（自定义封面）。
- FR12: 用户能按歌单名称或内容快速找到某个歌单。

### 播放控制与息屏

- FR13: 用户能播放、暂停、上一首、下一首当前歌单中的视频。
- FR14: 用户能选择单曲循环、列表循环或随机播放模式。
- FR15: 用户能在息屏或切到其他应用时继续听当前视频的音频（后台播放）。
- FR16: 用户能通过通知栏控制播放/暂停、上一首/下一首（酷狗/QQ 音乐风格）。
- FR17: 用户能在播放界面使用手势控制（如滑动切歌、双击暂停等）。

### 收藏与分类

- FR18: 用户能将视频标记为收藏（与「喜欢」合并为同一能力）。
- FR19: 用户能为视频添加多个标签并进行管理。
- FR20: 用户能在列表视图与标签视图之间切换以浏览内容。
- FR21: 用户能按标签、歌单、收藏、历史等维度筛选或进入相应内容列表。

### 搜索与导航

- FR22: 用户能按视频标题进行模糊搜索。
- FR23: 用户能查看与使用历史搜索记录。
- FR24: 用户能在搜索结果中看到与关键词匹配的高亮。

### 看与听

- FR25: 用户能在应用内正常观看视频画面。
- FR26: 用户能调节播放倍速（如 0.5x～2x）。
- FR27: 用户能使用画中画模式观看视频。
- FR28: 用户能设置睡眠定时（定时停止播放）。
- FR29: 用户能查看最近播放、继续观看、最近添加等入口并从中进入播放。

### 歌词

- FR30: 用户能关联本地歌词文件与视频并保存该关联（路径）。
- FR31: 用户能在播放时查看与当前视频关联的歌词内容。
- FR32: 用户能查看包含图片与文字等富文本格式的歌词。

### 基础信息与展示

- FR33: 用户能查看视频的时长、分辨率、文件大小等基础信息。
- FR34: 用户能以卡片式浏览视频时看到上述基础信息或缩略图。
- FR35: 用户能在播放控制区使用悬浮式控制按钮（或等效入口）。

### 数据持久化与存储

- FR36: 系统能持久化播放列表、歌单、收藏、标签、播放历史等数据（本地）。
- FR37: 用户能依赖本地数据在离线状态下使用上述所有与本地数据相关的能力。

### 权限与错误

- FR38: 系统能在需要访问本地视频时向用户请求或引导授予存储/媒体相关权限。
- FR39: 系统能在权限不足、路径失效或扫描失败时向用户给出可理解的提示，并在可行时提供重试或重新扫描入口。

## Non-Functional Requirements

### Performance

- **扫描与列表**：在典型设备上，对≤5000 个本地视频的扫描与列表展示应在用户可接受时间内完成（例如≤30s），且列表滚动无明显卡顿。
- **播放启动**：从用户选择播放到首帧/首声可感知，延迟应控制在较短时间内（例如≤3s，视文件与设备而定）。
- **切歌与通知栏**：上一首/下一首、通知栏控制等操作应在 1s 内生效，避免明显迟滞。

### Reliability

- **后台与息屏**：在息屏、切到其他应用、接听电话等场景下，音频播放应能持续进行；仅在用户主动停止或系统极端资源策略下才中断。
- **进程与数据**：应用被系统回收后再次打开，应能恢复上次播放进度（或在合理范围内），且歌单、收藏、标签等本地数据不丢失。

### Security & Privacy

- **当前**：数据仅存于本设备，无账号、无云端；自用场景下以「可直接使用」为前提，不在此阶段细化权限申请或隐私政策。
- **后期**：若引入服务端或上架分发，再补充数据保护、权限说明与隐私合规要求。

### 其他类别

- **Scalability**：不适用（本地、单人使用）。
- **Accessibility**：本阶段不单独设定无障碍指标；若有需求再在后续版本补充。
- **Integration**：不适用（无对外系统对接）。
