# 视频时长怎么读、进度条拖动怎么生效（通用说明）

下文按**原理**写：换用别的播放器项目（ExoPlayer、IJK、AVPlayer 等思路也类似），只要对照「数据从哪来、seek 依赖什么」，就能套到自己的工程里。不展开具体类名与字节偏移。

---

## 一、时长是怎么被「读出来」的（本质上是好几套独立信息源）

系统里并没有一个按钮叫「读时长」，而是：**不同 API 从文件或数据库里取出不同层级的信息**。有的只读表、有的读文件头、有的要扫完整结构、有的要等解码器就绪。得到的是**毫秒数**（或微秒再换算），再交给 UI 显示成 `mm:ss`。

### 1. 媒体库索引（例如 MediaStore）

- **读的是什么**：扫描媒体时写入数据库的一列「时长」字段，相当于**缓存的结论**。
- **特点**：快、不打开文件；但**没人保证一定对**——没扫到、fMP4 扫不出来、厂商 ROM 实现差异，都会是 **0 或空**。
- **和「小米相册 0:00」的关系**：若相册只信这一列或只信一次很浅的读取，就容易和真实可播文件不一致。

### 2. 框架自带的「元数据提取器」（Android 上常见是 MediaMetadataRetriever）

- **读的是什么**：按系统支持的方式打开文件或 Uri，解析**容器/轨道的元数据**，其中一项是「整段时长」。
- **特点**：对**传统 MP4（时长写在 moov 里）**通常够用。
- **常见坑**：
  - **`content://` 流**：有些机型必须用「文件描述符 + **已知可读长度**」去 `setDataSource`，否则内部无法定界，**直接返回 0**（部分小米等 ROM 上常见）。
  - **分片 MP4（fMP4）**：全局头里**没有**完整时长，时长分散在后面很多个小盒子里；Retriever **往往仍按老 MP4 假设去读**，结果仍是 **0**。

### 3. MediaExtractor（按轨道读 Format）

- **读的是什么**：选轨道后，从 **MediaFormat** 里取各轨的 `KEY_DURATION`（一般是微秒），再取**最大值**等策略合成「整片时长」。
- **特点**：比 Retriever 更贴近「解码器看到的轨道信息」；但对 fMP4 若格式里仍不写满整段时长，**也会是 0**。

### 4. 与播放器同源的「元数据检索」（例如 Media3 的 MetadataRetriever）

- **读的是什么**：走**和 Exo 类似的解封装路径**读轨道与格式信息，得到时长。
- **特点**：和实际播放更一致；仍可能输给「结构太怪」的 fMP4。

### 5. 分片 MP4：必须按**文件结构**把时长「算」出来（不是再调一个万能 API）

- **问题本质**：这类文件像「很多小段拼起来」，**没有**一个单独的「总时长字段」给上面那些 API 一把抓走。
- **做法（概念）**：顺序读 ISO BMFF 里的盒子：在 **moov** 里找到**视频轨的标识和时间基准**；再对每个 **moof** 里该轨的采样信息，把**每段采样时长按时间基准累加**。累加结果换算成毫秒，才是**真实可播长度**。
- **说明**：这是「按容器规范解析 + 算术累加」，不是去抠某几个 bit 的教程；实现上就是**流式读 box、识别类型、在 trun 等结构里取 duration/sample 相关字段再相加**。

#### 5.1 附录：`tfhd`（Track Fragment Header）里 **track_ID** 从哪读、读错会怎样

规范依据：**ISO/IEC 14496-12**（BMFF）里的 **TrackFragmentHeaderBox（`tfhd`）**。它是一个 **FullBox**：

| 在「`tfhd` 载荷」内的偏移（字节） | 字段 | 含义 |
|----------------------------------|------|------|
| 0 | `version` | 1 字节，一般为 0 |
| 1–3 | `flags` | 3 字节（大端组合成 24 bit），决定后面跟哪些**可选字段** |
| **4–7** | **`track_ID`** | **32 位无符号整数（大端）**，表示本 fragment 对应哪条轨 |
| **8 起** | 可选字段 | **顺序和长度由 `flags` 决定**，例如常见组合里会出现 `base_data_offset`（常为 64 bit）、`default_sample_duration` 等 |

**本仓库里 `Fmp4DurationProbe` 用的缓冲区是什么**：`forEachChildBox` 在父盒子载荷里按「size + type + 子载荷」切盒子，因此 `tfhd` 对应的那段 **`tfhdInner` 已经是去掉外层 8 字节 `size/type` 之后的「`tfhd` 盒子载荷」**，也就是 **直接从 FullBox 的 `version` 开头**。因此 **轨道号必须 `readU32(tfhdInner, 4)`**，与 `parseTfhdDefaultSampleDuration` 里「先读 `flags`，再从偏移 8 起按 flag 跳过可选字段」的写法一致（偏移 8 是 **可选字段区的起点**，不是 `track_ID`）。

**若误把 `track_ID` 当成偏移 8 去读**（例如 `readU32(tfhdInner, 8)`）：

- 偏移 8 处往往是 **`base_data_offset` 等高 32 位或可选字段的前 4 字节**。很多文件里这些值为 **0** 或与本轨无关的数。
- 解析结果常被当成「轨道 0」。而 **`moov` / `tkhd` 里视频轨的 `track_ID` 往往是 1**（或其它非 0 值）。
- 后果：在 `sumVideoUnitsInMoof` 里 **`parseTfhdTrackId(tf) != videoTrackId` 恒成立**，该 `moof` 下所有 **`trun` 都被跳过**，**分片时长累加恒为 0**。
- 与系统 API 的关系：若 **MediaMetadataRetriever / MediaExtractor / Media3** 某条链仍能给出时长，列表里仍会显示正常；若这些 API 对该文件也返回 0，**就只剩错误的探针**，表现为「全程拿不到时长」——这与「容器是不是 MP4」无关，而是**自研解析偏移错误**导致的假阴性。

**和 UI 上「时间显示怪怪的」有什么关系**：误读二进制字段**不会**稳定地变成某一种固定字符串（例如 `06:215`），但会间接造成：

- **累加为 0** → 列表/进度条总长 **0:00**，或退回其它信息源；
- **若再配合错误的 tick 累加或错误的 timescale**（把别的偏移上的字节当成 `duration`/`sample_count` 参与运算），可能得到**离谱的总毫秒数**，格式化成 `分:秒` 时会出现**异常位数、秒字段超过 59、或小数/毫秒位被错当成「第三位秒」** 等观感问题——本质是 **字节对齐错了，读到的不是时长语义**。

实现对照：见 `app/.../data/Fmp4DurationProbe.kt` 中 `parseTfhdTrackId` 的注释与 `parseTfhdDefaultSampleDuration`（从偏移 8 起按 `flags` 解析可选字段）。

### 6. 「真的没办法」时的最后一招：临时起一个播放器

- **读的是什么**：用**同一个引擎**打开同一个 Uri，**prepare 到就绪**，此时播放器内部的 **Timeline 已建好**，再读它对外的 **duration**。
- **特点**：最重、最慢，但能覆盖「只有完整解封装后才知道总长」的怪文件；一般要加**超时**，避免卡死。

### 7. 实际产品里怎么做

- **列表里先显示**：优先用数据库/缓存；为 **0 的条目**在后台再按上面 **2→3→4→5→6** 依次试，**谁先得到正数就用谁**。
- **开始播放时**：播放引擎自己的 **duration** 最可信；若仍是未知，再用**事先写入曲目的时长**（来自前面的探测）给进度条当总长，避免 UI 永远是 0:00。
- **进度条显示总长**：常见策略是——**播放器已给出有效时长就用播放器**；否则再退回**曲目上附带的那段探测时长**，避免两套数互相打架。

## 二、拖动进度条时，整体上发生了什么

### 1. 正常情况（大多数 MP4、本地可随机访问文件）

1. 用户拖动滑块，UI 根据 **0～100%** 和 **总时长（毫秒）** 算出目标时间 `T`。
2. UI 通过**媒体会话或播放器接口**发起 **seek 到 T**（本质是：请解码管线跳到时间 T 附近的关键帧，再解码显示）。
3. 底层 **demuxer 支持随机访问**（有索引或能按时间定位），seek **一次或几次**就稳定在新位置。
4. UI 周期性用 **当前播放时间** 刷新滑块；用户松手后也可短暂以目标时间对齐显示，避免和网络/缓冲抖动打架。

这就是「刚开始」最标准的做法：**只负责把目标时间告诉播放器**。

### 2. 什么时候拖动会「不灵」或「先跳回 0 再跳过去」

核心通常只有几类原因：

- **容器对 demuxer 表现为「不可 seek」**（例如只有逐字节顺序读、没有完整索引的封装）。
- **`content://` 管道读**：和真实磁盘文件行为不同，seek 容易失败或表现异常。
- **分片 / 怪异 MP4**：播放器先按「能播」打开，但**时间轴不完整**，`seekTo` 被忽略或落到错误位置。
- **会话层认为没有 seek 能力**：客户端直接禁用拖动，或发了 seek 也被丢弃。

### 3. 工程上常见的整体对策（与具体类名无关）

**思路 A：让数据源变成「可随机访问」**

- 把 `content://` **复制成应用私有的普通文件**再播，很多情况下 seek 会立刻正常很多。

**思路 B：换一封装（流复制重封装）**

- 用系统 **MediaExtractor + MediaMuxer**（或 FFmpeg）**不重编码**，只重新打包成**时间戳交错良好、渐进可 seek 的 MP4**，再交给同一套播放器。适合「能播但拖不动」的 fMP4。

**思路 C：需要跳转时「重开一路播放」**

- **停止当前实例 → 用新 Uri（或同一队列）重新 set 媒体列表 → prepare → 进入就绪状态后，再对底层播放器 seek 到 T**。  
- 原因：有些实现会忽略「带起始位置的 setMediaItem」，所以**就绪后再 seek**更稳。
- **注意**：对**本地**和**网络流**要区分——同一套「硬重载」在本地坏文件上可能反复从 0 起，要慎用；远程流有时反而只剩这一条路。

**思路 D：会话层与 UI 层「时长 / 可 seek」声明**

- 若播放器内核报「不可 seek」但你知道文件**其实能拖**（或你准备用 B/C 兜底），可以在**包装后的 Player**上对会话暴露「允许 seek」和「有效总时长」，否则系统通知栏、车载蓝牙等客户端会直接禁止拖动。  
- **真正执行跳转**仍由 A/B/C 在本地完成，避免只改 UI 无效。

**思路 E：预热**

- 开播后**稍等片刻**在后台做 A 或 B，让用户**第一次拖进度条时**已经是在「好 seek」的副本上，减少「先回 0 再跳」的观感。

**思路 F：UI 与异步任务对齐**

- 重开、重封装是**异步**的：松手 seek 后，进度条不要立刻用旧位置刷新覆盖；**换歌要取消**上一笔异步 seek，避免跳到别的歌上还执行旧目标。

## 三、和「相册显示 0:00」的对比（一句话）

相册若只依赖 **数据库一列**或 **Retriever 单次读全局头**，在 **fMP4** 和 **部分 content://** 上会系统性失败；要稳定，就需要 **Extractor / 播放器同源检索 / 分片结构累加 / 带长度的 FD** 等**多条路径**里至少一条成功。
