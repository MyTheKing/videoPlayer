编译成功。双缓存策略已实现：

读取逻辑（loadFromFileOrPrefs）：

先读 SP/Room（缓存）
缓存有数据 → 直接用，同时同步到文件
缓存无数据（升级/重装丢失）→ 从文件补回
写入逻辑（saveSettings / syncPlaylistsToFile）：

同时写缓存 + 文件，两边都正常处理
授权/换路径时（bidirectionalSync）：

缓存有数据 → 缓存覆盖文件（旧文件数据被覆盖）
缓存无数据 → 从文件恢复到缓存
触发时机：

启动时 PlaylistViewModel.init → bidirectionalSync()
授权存储权限后 → syncFromFile() → bidirectionalSync()
更换存储路径后 → syncFromFile() → bidirectionalSync()




核心逻辑 — 首次授权：缓存有内容 → 缓存覆盖存储；缓存无内容 → 读存储。路径切换：新文件夹有数据 → 清空缓存，读文件覆盖；新文件夹空 → 缓存写入文件。


切换存储文件，把数据进行迁移，如果迁移文件夹存在文件，不迁移，只使用文件夹内的数据，并且会把这个文件夹内的数据覆盖缓存中的数据。
也就是第一次 默认的路径会直接被缓存覆盖
但后面切换其他文件夹不会了。只会文件覆盖缓存


 场景	                        行为
启动（默认文件夹）	    缓存有数据 → 缓存覆盖文件；缓存无数据 → 文件补回缓存
授权存储权限	       同上（默认文件夹）
切换存储路径	       新文件夹有数据 → 文件覆盖缓存；新文件夹无数据 → 缓存写入文件

切换路径 = 切换「库」，优先使用库里已有的数据，不会把旧库的数据覆盖过去。

更改文件夹存储路径，可以读取不同文件夹数据，空文件夹是迁移。不是空的是读取过来 覆盖缓存 。但是当第一次授权的时候，如果缓存有内容，是直接覆盖存储的 ，没内容才是读取存储
